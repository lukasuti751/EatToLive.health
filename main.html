Vitality points

On Herbo, the curator credits vitality to contributors. You can spend vitality with a reason hash (e.g. for rewards or governance).

How vitality works

Vitality is an on-chain balance per address. The curator credits it after reviewing herb entries or for campaign participation. No token transfer; it is internal ledger state. Use spendVitality(amount, reasonHash) to deduct with a reason.

Contract config

            
            
            
Set your deployed Herbo contract address. Immutable roles: curator, treasury, wellnessKeeper (see contract).

          

          

            
Contract roles (immutable)

            
    
              
  • curator — Registers categories, credits vitality (single or batch), creates campaigns. Set once at deploy.
    • 
              
  • treasury — Receives donation fees from logHerb (optional wei) and swept treasury accum. Cannot be changed.
    • 
              
  • wellnessKeeper — Can remove herb entries (keeperRemoveEntry) and remove remedies. Used for moderation.
    • 
              
  • owner — Can pause the ledger, set vitalityPerEntry, set donationFeeBps, withdraw excess to treasury. Transferable via Ownable.
    • 
            

            
All constructor addresses are randomly generated per deployment and are not reused from other contracts.

Campaigns

        
On Herbo, campaigns are time-bound (start block, end block). The curator creates them; contributors can join entries to a campaign while it is active. Use joinEntryToCampaign(entryId, campaignId) from the contract.

        

          
How to join

          
You must be the contributor of the herb entry. The campaign must exist and the current block must be between start and end. Each entry can join at most one campaign.

        

      

    


    

      

        
About EatToLive.health

        
EatToLive.health is the web interface for the Herbo on-chain wellness ledger. Herbo lets contributors log herb entries (name, benefit, category hashes), attach wellness notes, create remedies linked to entries, and join campaigns. The curator credits vitality; the wellness keeper can remove entries when needed. All data is stored on the EVM chain you deploy to. This site is for education and interaction only; it does not provide medical advice.

        
FAQ

        

          
What is Herbo?

          
Herbo is a Solidity contract (EVM) that acts as a wellness ledger. Users log herb entries (with name, benefit and category hashes), link remedies to entries, and earn vitality points when the curator credits them. No token is transferred; vitality is on-chain state.

        

        

          
Why hashes instead of plain text?

          
Storing keccak256 hashes keeps the chain compact and allows off-chain storage of the actual names and benefits. Front-ends and indexers can map hashes to labels. It also keeps the contract language-agnostic.

        

        

          
Who is the curator?

          
The curator address is set at deployment (immutable). Only that address can register categories, credit vitality, and create campaigns. The contract owner can pause the ledger and set fee BPS and vitality rate.

        

        

          
Can I log a herb without paying?

          
Yes. Use logHerbFree(nameHash, benefitHash, categoryHash). The category must already be registered by the curator. Optional wei can be sent with logHerb; a fee percentage goes to the treasury.

        

        

          
What are remedies?

          
Remedies link a title (hash) to an existing herb entry. You can log a remedy with logRemedy(titleHash, herbEntryIdRef). Batch logging is also supported. Remedies can be updated or removed by the author or the wellness keeper.

        

        

          
Is this medical advice?

          
No. The herb guide and remedy ideas on this site are for general interest only. Always consult a qualified healthcare provider before using herbs for health purposes.

How to compute hashes for the Ledger

        
To log an herb you need bytes32 nameHash, benefitHash and categoryHash. Off-chain (e.g. in JavaScript with ethers):

        
// Example with ethers v6
import { ethers } from "ethers";
const nameHash = ethers.keccak256(ethers.toUtf8Bytes("Basil"));
const benefitHash = ethers.keccak256(ethers.toUtf8Bytes("Digestive support"));
const categoryHash = ethers.keccak256(ethers.toUtf8Bytes("Digestive"));
// Then call contract.logHerbFree(nameHash, benefitHash, categoryHash);

        
Categories must be registered on the contract by the curator before you can log entries in that category.

        

        
Remedy workflow on Herbo

        
First log one or more herb entries with logHerbFree. Then call logRemedy(titleHash, herbEntryIdRef) to link a remedy title to an entry. You can batch log remedies with batchLogRemedies. To update a remedy title use updateRemedyTitle(remedyId, newTitleHash); only the author can update. The wellness keeper can remove entries or remedies when needed. Campaigns are created by the curator with start and end blocks; contributors join entries via joinEntryToCampaign.

        
Herbo is designed for any EVM mainnet or testnet. Deploy once; curator, treasury and wellnessKeeper are fixed. Use a wallet that supports the chain you deployed to.

        

        
Glossary

        
    
          
  • Herb entry — A single on-chain record: contributor, nameHash, benefitHash, categoryHash, block, optional wei, active flag, optional noteHash.
    • 
          
  • Category — A registered bytes32 category hash with a label hash and entry count. Only the curator can register or update labels.
    • 
          
  • Remedy — A title hash linked to one herb entry (herbEntryIdRef). Used for recipes or combinations.
    • 
          
  • Vitality — On-chain balance per address, credited by the curator. Can be spent with a reason hash.
    • 
          
  • Campaign — Time-bound period (start block, end block). Entries can join at most one campaign while it is active.
    • 
          
  • Donation fee — When logging with optional wei (logHerb), a percentage (donationFeeBps) goes to treasury; the rest can be used per contract logic.
    • 
        

        
For full contract API see the Herbo Solidity source. Events include HerbLogged, VitalityCredited, RemedyLogged, CampaignCreated, EntryJoinedCampaign and others. Errors use the HRB_ prefix (e.g. HRB_EntryNotFound, HRB_NotCurator).
